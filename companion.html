How it's structured

CompanionAI is the central script. Everything else either calls its public command methods or subscribes to its events no script holds a direct reference to another.

  • CompanionAI.cs state machine, NavMesh movement, perception, all commands
  • CompanionInputHandler.cs maps keybinds to CompanionAI commands, listens to events for HUD feedback
  • BottleProjectile.cs physics projectile, parabolic arc, target leading, collision + break effects
  • LootSpot.cs scene loot node, weighted random table, depletion tracking
Things worth noting
  • Follow anchor placed behind the player along their smoothed velocity direction, not a fixed offset. Fixes the classic "companion locks in front of you" problem.
  • Action lock coroutine a shared bool blocks locomotion animator writes during one-shot clips (throw, react). Without this the companion slides into the throw pose mid-animation.
  • Loot table per LootSpot each spot has its own weighted table in the Inspector, with the global table as fallback. Designers tune individual spots without touching code.
  • Static events five C# Actions on CompanionAI. Input and HUD subscribe independently, no scene wiring needed.
Public Command API
  • CommandFollow() force return to Follow from any state
  • CommandHide() move to nearest hiding spot
  • CommandClear() exit Hiding or Fleeing, return to Follow
  • CommandThrowBottle() throw at nearest zombie, silent no-op if invalid
  • CommandScavenge() move to nearest loot spot
  • HealCompanion(float amount) add HP, clamped to maxHealth
  • GiveBottles(int count) add bottles, clamped to maxBottleCount
Events Fired
  • OnZombieSpotted Action<Transform>, fires when first zombie enters perception radius
  • OnCompanionDied Action, fires when health reaches 0
  • OnItemScavenged Action<string>, fires on loot pick completion, passes item name
  • OnBottleThrown Action<int>, fires on launch, passes remaining count
  • OnBottlesEmpty Action, fires when last bottle is used
Seven States

Transitions happen on player commands, proximity checks, and damage events not polled externally.

StateEntryBehaviourExit
Follow Default / CommandFollow() Trails player via velocity anchor. Idles and wanders locally when player is stationary. Any command, or damage while zombies are nearby
MovingToHide CommandHide() Sprints to nearest hiding spot at hideRunSpeed. Arrives within hideArrivalRadius → Hiding
Hiding Arrival at hide spot Crouches, immune to zombie detection. CommandClear() or CommandFollow()
Fleeing Hit by zombie + fleeZombieThreshold nearby Sprints to a sampled NavMesh point away from threat cluster, then crouches. CommandClear()
MovingToScavenge CommandScavenge() Walks to nearest LootSpot. Aborts if zombie enters alertScanRadius. Arrives at spot → Scavenging; zombie spotted → Follow
Scavenging Arrival at LootSpot Waits scavengeDuration seconds, then calls LootSpot.Loot(). Timer expires → Follow
Dead Health reaches 0 Terminal. No transitions out. OnCompanionDied fires. None
Follow Anchor

Each frame, the player's position delta is blended into _smoothVelocityDir via Vector3.Slerp. The companion targets a point behind the player along that direction, with a lateral offset to keep her out of the shooting line.

During sprinting, the lateral offset fades to zero and she trails directly behind. If she gets too close, a push-back kicks in. The result is she sits just behind your shoulder close enough to feel present, out of the way when it matters.

behindDir = -_smoothVelocityDir moveRight = Cross(Vector3.up, _smoothVelocityDir) anchor = player.pos + behindDir * activeDistance + moveRight * activeLateral
Bottle Arc

CalculateArcVelocity() works out the exact initial velocity to hit the target in a single parabolic arc. The peak is measured above whichever endpoint is higher so it clears geometry regardless of the height difference between thrower and target.

peakY = Max(start.y, end.y) + arcHeight timeToPeak = Sqrt(2 * riseH / gravity) timeToLand = Sqrt(2 * fallH / gravity) velXZ = dispXZ / totalTime velY = gravity * timeToPeak

Rigidbody drag is forced to 0 before launch any drag makes the bottle fall short. Angular drag (0.05) stays on for a natural spin.

Throw Safety

CommandThrowBottle() does nothing if the companion is dead, hiding, out of bottles, on cooldown, or no zombie is in range. If all checks pass: play the throw animation, lock locomotion so she doesn't slide mid-throw, spawn the projectile, decrement bottle count.

Collision & Startle

Both OnCollisionEnter and OnTriggerEnter call the same handler, which deals damage and calls ZombieAI.NotifyStartle() treated as a mid-priority distraction. Break particles and audio are detached before the GameObject is destroyed so they finish playing after it's gone.

Loot
  • Each LootSpot has a weighted table a weight-5 item is five times as likely as weight-1. CommandScavenge() finds the nearest non-depleted spot via OverlapSphere, waits the scavenge timer, then calls Loot().
  • Any zombie entering the scan radius during transit cancels the scavenge.
  • In the Scene view, LootSpots show a wire cube and remaining picks ("2/3" or "DEPLETED") useful for level design without selecting every spot.
Perception

CR_ZombieAlertScan() runs every 0.8s. It does an OverlapSphere, filters by line-of-sight from eye height, and keeps a HashSet of known zombies. Dead ones are pruned each pass. Perception doesn't auto-trigger hiding it drives voice barks and the flee threshold check only.

Animator

DriveAnimator() runs at the end of every Update and sets locomotion bools but only if the action lock is inactive, so a throw or react clip won't get interrupted. One setup requirement: the 'React' Animator state needs its Tag set to 'React', and 'Throw' to 'ThrowBottle'. The lock coroutine polls the current tag to know when the clip has finished. Without those tags, the companion freezes after the first hit or throw.
